git merge vs rebase

My personal/public journal

If you've ever worked on a team with more than one developer, on any project with complexity, there is a 99.9999% chance you've experienced merge conflicts. These conflicts have plagued every developer since gits conception. Ever tried to pull a branch and recieved the dreaded "local branch is behind...."? Ever just said f*ck it and used the old trusty "--hard"? Well maybe that last one is me, but regardless of how many years you've in this space, these issues are common.

Where I currently work, we don't have any strict policies around our git workflows. Want to merge? Sure! Want to rebase? Go for it! Want to do a hard reset on main and force push? No proble... well actually no, please don't ever do that. But ultimately, it's up to us on deciding how to resolve conflicts. Through this freedom, and the weight of trying not to permanently destroy our code base, I've learned become quite fimiliar with gits conflict resolution tools.
